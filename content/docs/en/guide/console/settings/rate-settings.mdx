Ratio settings are a core part of AI Router's internal cost-accounting system. By configuring different ratios you can flexibly control how usage of various models and user groups is converted into quota consumption — useful for internal cost allocation, budget control, or differentiated billing tiers.

## Ratio System Overview

AI Router uses a three-tier ratio system to calculate quota consumption:

1. **Model Ratio** — the base billing multiplier for a given model.
2. **Completion Ratio** — an additional adjustment applied to output tokens.
3. **Group Ratio** — a differentiated multiplier for different user groups.

### Relationship between Quota and Ratios

In AI Router, ratios are the parameters used to compute quota consumption. **Quota** is the internal accounting unit of the system: all API calls are ultimately converted into quota points and deducted from a user's balance.

Quota unit conversion:

- 1 USD ≈ **(conversion rate TBD / 待补)** quota points
- Quota points are the basic unit for all internal accounting.
- User balances and consumption records are tracked in quota points.

<Callout type="info">
The USD-to-quota conversion rate is an operational decision and is **not yet finalized** for this demo. Configure it to a value that suits your deployment before exposing billing to end users.
</Callout>

### Quota Calculation Formulas

#### Pay-as-you-go models (token-based)

```
quota = (input_tokens + output_tokens × completion_ratio) × model_ratio × group_ratio
```

#### Per-call billing models (fixed price)

```
quota = model_fixed_price × group_ratio × quota_unit
```

#### Audio models (handled automatically by AI Router)

```
quota = (text_input_tokens
         + text_output_tokens × completion_ratio
         + audio_input_tokens × audio_ratio
         + audio_output_tokens × audio_ratio × audio_completion_ratio)
        × model_ratio × group_ratio
```

#### Pre-consumption and post-consumption

AI Router uses a two-phase billing mechanism:

1. **Pre-consumption** — before the call, an estimated quota amount is pre-deducted based on the estimated token count.
2. **Post-consumption** — after the call completes, quota is recalculated from the actual token count.
3. **Difference adjustment** — if the actual consumption differs from the pre-consumption, the system automatically corrects the user's balance.

```
pre_consumed = estimated_tokens × model_ratio × group_ratio
actual       = actual_tokens × model_ratio × group_ratio
adjustment   = actual - pre_consumed
```

## Completion Ratio Settings

The completion ratio applies extra billing to output tokens, mainly to balance the cost difference between input and output for models where output is more expensive than input.

### Setup instructions

- The completion ratio affects only output-token billing.
- A value of **1** means output tokens are billed the same as input tokens.
- Greater than 1 → output tokens cost more; less than 1 → output tokens cost less.

<Callout type="info">
Per-model completion-ratio defaults depend on each upstream model's published input/output pricing, which is **not finalized in this demo (待补)**. Set conservative values and adjust based on the upstream provider's actual rates.
</Callout>

## Models Without Ratio Settings

For models that have no ratio configured, AI Router can:

1. **Self-use mode** — fall back to a default ratio (**default value TBD / 待补**).
2. **Billing mode** — return a "ratio or price not configured" error rather than guessing a price.
3. **Detection** — surface unconfigured models in the management interface so they can be filled in.

## Upstream Ratio Synchronization

AI Router can help keep local ratios aligned with upstream channels:

- Sync only legally authorized, publicly available price/model metadata from upstream.
- Batch-update local ratio configurations.
- Keep local entries roughly aligned with upstream rates.
- Allow manual adjustment and override at any time.

## Frequently Asked Questions

<Accordions>
<Accordion title="How do I set a ratio for a new model?">
Add the model through the visual editor or directly in the JSON configuration. Start with a conservative ratio and adjust it based on actual usage and the upstream provider's real pricing.
</Accordion>

<Accordion title="How do group ratios take effect?">
A group ratio is multiplied with the model ratio when computing quota consumption — so a user's effective multiplier is roughly `model_ratio × group_ratio` (with any user-specific override taking precedence).
</Accordion>

<Accordion title="What is the completion ratio for?">
It balances the cost difference between input and output tokens. For models whose output is significantly more expensive than input, raise the completion ratio so output tokens are billed proportionally more.
</Accordion>

<Accordion title="How do I batch-set ratios for similar models?">
Use the visual editor's batch operations, or add several entries at once in the JSON configuration.
</Accordion>
</Accordions>
